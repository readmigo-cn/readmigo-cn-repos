## 决策背景

### 背景：两个场景，一个公司主体？

米果智读（Readmigo CN）需要选择运营主体，具体选项：

**方案 A**：继续沿用 **北京瑞光科技有限公司**（既有主体，已运营海外版 Readmigo）  
**方案 B**：新设 **北京某某科技有限公司**（国内独立子公司）

**核心问题**：
- 海外版 Readmigo 由北京瑞光科技开发，2026 年在海外市场运营
- 国内版米果智读也需要一个法律主体开发 / 运营
- 两个版本的数据、用户、服务是**完全隔离**的（合规要求）
- 一家公司能否同时运营两个隔离的产品线？答案：**能**，但需要清晰的治理结构

### 维度 2：数据隔离与合规风险

#### 方案 A：沿用北京瑞光科技

**技术架构**：
```
┌──────────────────────────────────┐
│      北京瑞光科技有限公司         │
│                                  │
│  业务 A：Readmigo 海外版          │
│  ├─ 用户数据 → Neon/Cloudflare   │
│  ├─ LLM → OpenAI / Claude         │
│  └─ 服务器 → 海外 CDN             │
│                                  │
│  业务 B：米果智读（国内版）       │
│  ├─ 用户数据 → 华为云 GaussDB   │
│  ├─ LLM → DeepSeek / 文心         │
│  └─ 服务器 → 华为云 CDN          │
│                                  │
│ ⚠️ 治理模型：                    │
│  - 法律合同分离（两套 ToS）       │
│  - 技术栈完全隔离（两个 DB）      │
│  - 运营团队分离（OKR 独立）       │
└──────────────────────────────────┘
```

**合规风险评估**：

| 合规项 | 风险 | 缓解措施 |
|------|------|--------|
| **个人信息保护法（PIPL）** | 低 | 两套独立隐私政策 + 分离的数据库 + 清晰告知用户 |
| **生成式 AI 备案** | 低 | 米果智读作为"下游应用"独立备案，不与海外版混淆 |
| **内容审核** | 低 | 国内版单独建立审核制度；海外版内容政策不影响国内版 |
| **ICP 备案** | 中 | 需提交"企业多产品备案"说明（说明两个版本服务不同用户） |
| **数据安全** | 低 | 华为云 + Neon/Cloudflare 地理位置完全分离，审计日志独立 |

**结论**：**合规风险可控**，技术隔离充分。关键是做好治理文档和合同分离。

#### 方案 B：新设独立子公司

**治理架构**：
```
┌──────────────────────────────┐
│   北京瑞光科技（母公司）      │
│                              │
│   ├─ 业务 A：Readmigo 海外   │
│   │                          │
│   └─ 子公司：米果智读有限公司 │
│       └─ 业务 B：国内版       │
└──────────────────────────────┘
```

**合规优势**：
- ✅ 法律实体分离，天然合规隔离
- ✅ 无需解释"为什么一家公司运营两个版本"
- ✅ ICP 备案、生成式 AI 备案 更直观（独立主体）
- ✅ 独立财务报表 / 估值 / 融资（如有需要）

**合规成本**：
- 新设企业需额外合规（**2-3 周时间**）
- 管理复杂度上升（两个税号、两套制度）
- 初期工商 + 税务 + 开户 成本约 **¥5,000-10,000**

**结论**：合规最干净，但**延期 2-3 周 + 增加成本**

### 维度 3：财务与融资

#### 方案 A：沿用北京瑞光科技

**优点**：
- ✅ 单一报表，财务简洁
- ✅ 运营成本共享（办公、IT 等基础设施）
- ✅ 两个业务线的 ARR 合并显示，总收入更显眼

**缺点**：
- ❌ 难以单独估值国内版（风险投资方希望看到国内版独立 P&L）
- ❌ 如果将来需要独立融资 / IPO 国内版，需要"拆分"（复杂）

**融资建议**：
- 如果 2026-2027 年无独立融资计划 → 方案 A 足够
- 如果 2027-2028 年计划融资国内版 → 现在新设子公司成本更低

---

#### 方案 B：新设独立子公司

**优点**：
- ✅ 独立 P&L，便于单独融资 / 估值
- ✅ 更容易吸引国内投资方（VCs 更看好"国内独立团队"的治理）

**缺点**：
- ❌ 初期管理复杂度上升
- ❌ 需要独立团队 CFO 或财务（前期不必要）

## 推荐方案

### **✅ 推荐：方案 A - 沿用北京瑞光科技有限公司**

**决策理由（权重排序）**：

1. **时间成本最低**（1.5 周 vs 3 周）
   - 米果智读有 12 周的 AI 备案窗口（deadline 2026-07 中旬）
   - 新设子公司会吃掉 2 周 = 只剩 10 周，风险上升
   - **建议 Day 1 启动，新设会延期到 Week 3 才启动**

2. **经营范围完全覆盖**
   - "AI 应用软件开发"、"数据处理" 已在营业执照
   - 无需工商变更，即刻可用

3. **合规隔离完全充分**
   - 两套数据库（华为云 + Neon）
   - 两套 ToS / 隐私政策
   - 两套 LLM 合作协议
   - 合规评估报告证明隔离
   - **法律隔离 = 一家公司，治理边界清晰**

4. **财务简洁，融资友好**
   - 如果 2027 年需融资 → 可从单一 P&L 中拆分"国内版"的 metrics 呈现
   - VCs 更看重"产品、用户、收入"而非"法律实体"

5. **运营成本共享**
   - 办公地、HR、财务后端共享，初期成本更低

## 风险评估

### 风险 1：北京瑞光科技有不良信用记录（概率 5%）

**触发条件**：工商信用查询发现欠税 / 行政处罚

**应对**：
1. 立刻启动方案 B（新设子公司）
2. 时间延期 2 周 → AI 备案延期到 Week 7（仍在可接受范围）
3. 成本增加 ¥15,000

**缓解**：
- 2026-05-02 日立即启动信用查询，**第一时间发现**
- 预留备选新公司设立的律师合同（待签）

---

### 风险 2：监管机构拒绝"一家公司两个隔离服务"（概率 15%）

**触发条件**：网信办 / 工信部 在 ICP 备案或 AI 备案环节驳回"不支持一企多产品"

**应对**：
1. 即刻启动方案 B（新设子公司）
2. 重新提交备案申请（以子公司名义）
3. 时间延期 2-3 周，但 AI 备案 still 在 12 周窗口内

**缓解**：
- 2026-05-05 的网信办沟通中**提前问清楚**
- 如果反馈不明朗 → 主动提案"我们可以新设子公司"（展现合规意愿）

---

### 风险 3：律师意见：隔离方案不够充分（概率 10%）

**触发条件**：外部律师认为技术隔离 + ToS 隔离不足，必须法律实体分离

**应对**：
1. 听从律师意见，启动方案 B
2. 法律合规责任由律师承诺 + 保险覆盖

**缓解**：
- 选择有"生成式 AI 备案"经验的律师（不选通用律师）
- 获取律师的"合规隔离可行性意见书"

---

### 风险等级总结

| 风险 | 概率 | 影响 | 累计延期 | 缓解方式 |
|-----|------|------|--------|--------|
| **信用不良** | 5% | 强制方案 B | 2 周 | 提前查询 |
| **监管驳回** | 15% | 启动方案 B | 2-3 周 | 提前沟通 |
| **律师否决** | 10% | 启动方案 B | 2-3 周 | 选对律师 |
| **综合风险** | ~25% | 延期方案 B | 最差 3 周 | — |

**建议**：假设有 25% 的风险切到方案 B，**预留 1 周缓冲**，即：

```
最坏情况时间表：
Week 1  → 决策 + 信用查询 + 法务沟通（如有风险转方案 B）
Week 2-3 → 方案 B：新设子公司 + ICP 备案
Week 4  → 补偿周期 + 备案审核
Week 5+ → AI 备案启动（still 在 12 周 deadline）
```
