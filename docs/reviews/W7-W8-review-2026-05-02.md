### 6 个 Service 对比

| Service | 模式 | 同步队列 | 缓存 | 错误处理 | 评价 |
|---------|------|--------|------|---------|------|
| ProgressSync | Singleton | debounce 5s + 指数退避 | ReadingStore | 网络监听 + 冲突回调 | ✓ 完整 |
| ReadingSession | Singleton | tick 30s | 内存 | 会话超时逻辑 | ✓ 完整 |
| HighlightService | Singleton | debounce 5s + 指数退避 | 内存 LRU | 失败入队 + 持久化 | ✓ 完整 |
| NoteService | Singleton | debounce 5s + 指数退避 | 内存双轨 | 失败入队 | ✓ 完整 |
| TranslateService | Singleton | N/A (sync+stream) | LRU 50 | 缓存命中 fallback | ⚠️ 缺 timeout |
| TtsService | Singleton | N/A (stream) | N/A | mock 限制 | ⚠️ 缺 abort 策略 |

### DRY 违规与改进建议

**同步队列模板重复** (ProgressSync / HighlightService / NoteService)

三个 service 的同步队列逻辑几乎相同:
```ets
// 都有这些
private pending: T[] = [];
private debounceTimer: number = -1;
private retryTimer: number = -1;

private scheduleDebounce() { ... }  // 完全相同
private scheduleBackoffRetry() { ... }  // 完全相同
private persistQueue() { ... }  // 几乎相同
```

**改进方案（W10）**:
```ets
abstract class BaseSyncService {
  protected pending: PendingOp[] = [];
  protected debounceTimer: number = -1;
  protected retryTimer: number = -1;

  protected scheduleDebounce(callback: () => Promise<void>) { ... }
  protected scheduleBackoffRetry(callback: () => Promise<void>) { ... }
  protected async flushImpl(handler: (op) => Promise<void>) { ... }
  abstract persistKey(): string;
}

// 使用方
class HighlightServiceImpl extends BaseSyncService {
  async flush() {
    return this.flushImpl((op) => this.runOp(op));
  }
}
```

**单例模式生成器缺失**

每个 service 都手写:
```ets
static instance: ServiceType | null = null;
static getInstance(): ServiceType { ... }
```

可以添加工具函数减少重复（选项，不强制）。

### 网络与错误处理

**评分**: 7/10

- ✓ ProgressSync 的网络监听完整（onNetAvailable / onNetUnavailable / onNetLost）
- ✓ 三个同步队列都有指数退避（1s / 2s / 4s / 8s / 16s / 32s / 60s）
- ✓ 崩溃恢复依赖持久化队列，启动时自动 restore

   ```ets
   cancel(): void {
     if (this.timer !== -1) {

   }
   ```

## 4. SSE 与流式渲染质量

### ExplainCard 的 SSE 实现

**文件**: `components/ExplainCard.ets` (530 LOC)

**流程**:
1. 接收 `word + context`
2. 触发 `AiApi.explainWordStream`（内部调 SseClient）
3. onMessage 中累积 raw 文本 → `@State streamedText`
4. 三态渲染: skeleton → streaming → structured (parseExplainData)

### 问题分析

**🔴 Critical: 流式解析的边界缺失**

```ets
// ExplainCard.ets 约 120 行
onMessage: (raw: string) => {
  this.streamedText += raw;  // 无限累积
  // 缺少 incremental parse：不知道何时"块"完成
}
onDone: () => {
  this.parsed = parseExplainData(this.streamedText);  // 一次性解析整个累积文本
}
```

**风险**:
- 如果服务端错误（如发送非 JSON），UI 永久卡在 streaming 态
- 无法实时显示"部分解析结果"（当前等全部到达再一次解析）
- 后续 NoteService.addVocab() 依赖 parsed，若为 null 则无法保存

**改进方案**:

```ets
// 尝试 incremental parse（行迭代）
onMessage: (raw: string) => {
  this.streamedText += raw;
  const lines = this.streamedText.split('\n');
  // 保留最后一个不完整行
  for (let i = 0; i < lines.length - 1; i++) {
    if (this.tryParseLine(lines[i])) {
      // 局部解析成功，更新 parsed
    }
  }
}
```

**🟡 High: 重连与取消策略不足**

```ets
// SseClient.stream() 返回 stop() 函数
const stop = SseClient.stream('/ai/explain-word', {...}, handlers);

// 但 ExplainCard 中：
// 1. 未暴露 stop，无法让上层中止流
// 2. 网络中断后无自动重试
```

**改进**:

```ets
@State private stopStream: (() => void) | null = null;

private startExplainStream() {
  this.stopStream = SseClient.stream(..., {
    onMessage: ...,
    onError: (err) => {
      if (this.retries < 3) {
        setTimeout(() => this.startExplainStream(), 1000 * this.retries);
      } else {
        this.errMsg = '解释生成失败，请重试';
      }
    }
  });
}

onClose() {
  this.stopStream?.();
}
```

**⚠️ SseClient 本身的问题**

```ets
req.on('dataReceive', (chunk: ArrayBuffer) => {
  if (aborted) return;
  buffer += decoder.decode(chunk, { stream: true });
  // 没有 chunk 大小限制，buffer 可能无限增长
})
```

**改进**: 限制 buffer 大小（如 1MB）。

### 性能影响

**频繁 re-render 风险**:

ExplainCard 中每收到一个 SSE chunk 就触发 `@State streamedText` 更新：
- 如果服务端发送速度快（如 100ms 一次），会触发 100+ 次 UI rebuild
- 合并 chunk 在一定时间内（如 200ms）再更新，减少 rebuild

### TTS + Highlight + AI 并发运行

**场景**: 用户启动 TTS 朗读，同时拖拽选词触发 ExplainCard 流式翻译

**当前行为**:
- TTS: MockTtsHandle 的 timer 每 10ms 一次 onProgress，更新 `ttsCharIndex`
- Highlight: SentenceHighlight 根据 `ttsCharIndex` 计算高亮矩形
- ExplainCard: 每收到 SSE chunk 就更新 `streamedText`

**问题**:
- `ttsCharIndex` 变化 → SentenceHighlight rebuild → getRectsForRange 调用
- getRectsForRange 内部遍历 page.lines，计算复杂度 O(lines)
- 若同时触发 ExplainCard rebuild，主线程可能卡顿 50-100ms

**改进建议**:
1. TTS 的 onProgress 添加 debounce（如 50ms 一次）
2. SentenceHighlight 的 getRectsForRange 缓存前两次结果，避免重复计算
3. W9 真机测试 60 FPS 是否稳定

### LRU Cache 命中率监控

**TranslateService** 有 LRU(50)，但缺命中率统计：

```ets
// 建议补充
private hits: number = 0;

get(key: string): TranslateResult | undefined {
  const v = this.map.get(key);
  if (v) this.hits++;
  else this.misses++;
  ...
}

getStats(): { hits, misses, ratio } { ... }
```

这样便于后续 W10 做性能优化决策（cache size 是否够）。
