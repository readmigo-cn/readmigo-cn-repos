### 1.2 AI Quota Upsert Race Condition

**Location**: `server-cn/src/modules/ai/ai.service.ts:197-228`

**Problem**: The `_incrementQuota()` method uses `orUpdate()` with a fallback to manual read-then-write. However:
1. The `setParameter('count', () => 'ai_quotas.count + 1')` syntax is **database-dialect-dependent** (PostgreSQL-specific raw SQL)
2. The fallback doesn't use a transaction, so two concurrent requests can both read `count=5`, then both write `count=6` (should be `7`)
3. If the dialect doesn't support `orUpdate()`, the catch silently swallows the error and executes fallback—but there's no retry loop

**Current Code**:
```typescript
async _incrementQuota(userId: string, kind: AiConversationKind): Promise<void> {
  // ... 
  await this.quotaRepo
    .createQueryBuilder()
    .insert()
    .into(AiQuotaEntity)
    .values({ ... })
    .orUpdate(['count', 'resetAt'], ['userId', 'period', 'kind'])
    .setParameter('count', () => 'ai_quotas.count + 1')  // ← PostgreSQL specific
    .execute()
    .catch(async () => {
      // ← Fallback doesn't use transaction
      const existing = await this.quotaRepo.findOne({ where: { userId, period: 'daily', kind } });
      if (existing) {
        if (existing.resetAt <= new Date()) {
          await this.quotaRepo.update(existing.id, { count: 1, resetAt, limit });
        } else {
          await this.quotaRepo.increment({ id: existing.id }, 'count', 1);  // ← OK but can race
        }
      } else {
        await this.quotaRepo.save(...);
      }
    });
}
```

**Suggested Fix**:
```typescript
async _incrementQuota(userId: string, kind: AiConversationKind): Promise<void> {
  const resetAt = this._dailyResetAt();
  const limit = QUOTA_DAILY_LIMITS[kind] ?? 50;

  // Use transaction to ensure atomicity
  const queryRunner = this.quotaRepo.manager.connection.createQueryRunner();
  await queryRunner.connect();
  await queryRunner.startTransaction();
  
  try {
    // Use dialect-agnostic increment
    const existing = await this.quotaRepo.findOne({ 
      where: { userId, period: 'daily', kind },
      lock: { mode: 'pessimistic_write' }  // Row-level lock
    });
    
    if (existing) {
      if (existing.resetAt <= new Date()) {
        existing.count = 1;
        existing.resetAt = resetAt;
      } else {
        existing.count += 1;
      }
      await this.quotaRepo.save(existing);
    } else {
      await this.quotaRepo.save(
        this.quotaRepo.create({ userId, period: 'daily', kind, count: 1, limit, resetAt })
      );
    }
    await queryRunner.commitTransaction();
  } catch (err) {
    await queryRunner.rollbackTransaction();
    throw err;
  } finally {
    await queryRunner.release();
  }
}
```

**Rationale**: Prevents quota double-counting in high-concurrency scenarios (multi-device sync). This is critical for billing accuracy.

### 1.3 Refresh Token Hash Mismatch Logic Error

**Location**: `server-cn/src/modules/auth/auth.service.ts:200-223`

**Problem**: The refresh token validation compares the token **hash stored in the database against the hash of the incoming token**. However, there's a subtle issue: if the incoming token is passed incorrectly (e.g., the client sends the token plaintext, but the server stores a hash of it), the comparison will always fail on re-use.

The logic is actually **correct**—the server stores `hash(token)` and compares `hash(incomingToken) === storedHash`. But the **session deletion happens before validating the user status**, creating a window where a 401 is returned but the session is already deleted, so retry won't work.

**Current Code**:
```typescript
async refresh(refreshToken: string, deviceId: string): Promise<TokenPair> {
  // ... verify JWT signature ...
  const session = await this.sessions.findOne(...);  // ← Found
  if (!session) throw new UnauthorizedException('session_not_found');
  
  if (session.refreshTokenHash !== this._hashRefreshToken(refreshToken)) {
    throw new UnauthorizedException('refresh_token_mismatch');  // ← Session still in DB
  }

  const user = await this.users.findOne({ where: { id: payload.sub } });
  if (!user || !user.isActive) throw new UnauthorizedException('user_inactive');  // ← Late check!

  // Delete old session
  await this.sessions.delete(session.id);  // ← Deletion before user check causes race

  return this._generateTokens(user, session.deviceId, session.deviceName ?? undefined);
}
```

**Suggested Fix**:
```typescript
async refresh(refreshToken: string, deviceId: string): Promise<TokenPair> {
  const refreshSecret = this.cfg.get<string>('JWT_REFRESH_SECRET') ?? 'dev-refresh-secret-change-me';

  let payload: { sub?: string };
  try {
    payload = jwt.verify(refreshToken, refreshSecret) as { sub?: string };
  } catch {
    throw new UnauthorizedException('invalid_refresh_token');
  }

  if (!payload.sub) throw new UnauthorizedException('invalid_refresh_token');

  // Check user status FIRST (before session lookup)
  const user = await this.users.findOne({ where: { id: payload.sub } });
  if (!user || !user.isActive) throw new UnauthorizedException('user_inactive');

  // Then check session
  const session = await this.sessions
    .createQueryBuilder('s')
    .addSelect('s.refreshTokenHash')
    .where('s.userId = :userId', { userId: payload.sub })
    .andWhere('s.deviceId = :deviceId', { deviceId })
    .andWhere('s.expiresAt > NOW()')
    .getOne();

  if (!session) throw new UnauthorizedException('session_not_found');

  if (session.refreshTokenHash !== this._hashRefreshToken(refreshToken)) {
    throw new UnauthorizedException('refresh_token_mismatch');
  }

  // Now delete old session (user is confirmed active)
  await this.sessions.delete(session.id);

  return this._generateTokens(user, session.deviceId, session.deviceName ?? undefined);
}
```

**Rationale**: Ensures authorization checks are completed before any state modification.

### 2.1 Reading Stats N+1 Query (Minor)

**Location**: `server-cn/src/modules/reading/reading.service.ts:114-134`

**Problem**: `getReadingStats()` first fetches **all progress rows** in memory (`await this.progress.find({ where: { userId } })`), then loops to calculate stats. If a user has 1000 books, this loads 1000 rows just to compute totals.

**Impact**: Unnecessary memory pressure on large bookshelves; SQL aggregation is faster.

**Suggested Fix**:
```typescript
async getReadingStats(userId: string, period: string): Promise<ReadingStats> {
  // Aggregate in SQL
  const stats = await this.progress
    .createQueryBuilder('p')
    .select('SUM(p.totalReadMinutes)', 'totalReadMinutes')
    .addSelect('COUNT(CASE WHEN p.percentComplete >= 0.99 THEN 1 END)', 'booksFinished')
    .where('p.userId = :userId', { userId })
    .getRawOne();

  const sessionCount = await this.sessions
    .createQueryBuilder('s')
    .where('s.userId = :userId', { userId })
    .andWhere('s.endedAt IS NOT NULL')
    .getCount();

  return {
    totalReadMinutes: stats?.totalReadMinutes ?? 0,
    booksFinished: stats?.booksFinished ?? 0,
    totalSessions: sessionCount,
    avgSessionMinutes: sessionCount > 0 
      ? Math.round(((stats?.totalReadMinutes ?? 0) / sessionCount) * 10) / 10 
      : 0,
    period,
  };
}
```

### 2.3 No Transient Error Retry in ArkTS HttpClient After Refresh

**Location**: `harmony-app/entry/src/main/ets/service/HttpClient.ets:154-162`

**Problem**: When a 401 is caught, the client calls `HttpClient.tryRefresh()` and **retries the original request once**. However, if the retry itself fails with 5xx, there's no exponential backoff—it just throws.

**Current Code**:
```typescript
if (apiErr.statusCode === 401 && !ctx.options.skipRefresh) {
  const refreshed = await HttpClient.tryRefresh();
  if (refreshed) {
    ctx.headers['Authorization'] = `Bearer ${refreshed}`;
    return HttpClient.execute<T>(path, ctx, 0); // ← Retry with 0 retries left
  }
  HttpClient.handleAuthFailure();
  throw apiErr;
}
```

**Impact**: If the refresh succeeds but the retry fails with 5xx, the user sees a hard error instead of exponential backoff. The `retriesLeft` is set to 0, which means **no retry logic is applied**.

**Suggested Fix**:
```typescript
if (apiErr.statusCode === 401 && !ctx.options.skipRefresh) {
  const refreshed = await HttpClient.tryRefresh();
  if (refreshed) {
    ctx.headers['Authorization'] = `Bearer ${refreshed}`;
    return HttpClient.execute<T>(path, ctx, 1); // ← Allow 1 more retry for the refreshed request
  }
  HttpClient.handleAuthFailure();
  throw apiErr;
}
```

### 2.4 SSE Stream Error Handling Missing in AiService

**Location**: `server-cn/src/modules/ai/ai.service.ts:378-391`

**Problem**: The `translateSentenceStream()` method doesn't handle provider timeouts or partial stream failures. If the LLM provider times out at 10 seconds into a 30-second response:
1. The `for await` loop breaks (provider's timeout)
2. The `finally` block calls `sseSink.complete()`, signaling to the client that the stream is done
3. But the partial response is saved to the database as complete

The client sees an incomplete translation without knowing it timed out.

**Current Code**:
```typescript
try {
  for await (const chunk of stream) {
    if (chunk.content) {
      full += chunk.content;
      sseSink.next({ data: JSON.stringify({ delta: chunk.content }) });
    }
    if (chunk.done) break;
  }
  sseSink.next({ data: JSON.stringify({ done: true }) });
} catch (err) {
  sseSink.next({ data: JSON.stringify({ error: (err as Error).message }) });
} finally {
  sseSink.complete();
}
```

**Suggested Fix**:
```typescript
let isError = false;
try {
  for await (const chunk of stream) {
    if (chunk.content) {
      full += chunk.content;
      sseSink.next({ data: JSON.stringify({ delta: chunk.content }) });
    }
    if (chunk.done) break;
  }
  if (full.length === 0) {
    throw new Error('Empty response from provider');
  }
  sseSink.next({ data: JSON.stringify({ done: true }) });
} catch (err) {
  isError = true;
  sseSink.next({ data: JSON.stringify({ error: (err as Error).message }) });
} finally {
  sseSink.complete();
  // Only persist if stream completed successfully
  if (!isError && full.length > 0) {
    const cacheKey = this._buildCacheKey('sentence-translate', ...);
    await this._writeCache(cacheKey, 'sentence-translate', full, provider.name);
    const conv = await this._persistConversation(userId, 'sentence-translate', { ... });
    await this._persistMessage(conv.id, 'user', dto.sentence, null, null, 0);
    await this._persistMessage(conv.id, 'assistant', full, provider.name, null, 0);
  }
}
```

### 2.5 Hardcoded Dev Secret in JWT Guard

**Location**: `server-cn/src/modules/auth/auth.service.ts:19`, `jwt-auth.guard.ts:19`

**Problem**: Both `AuthService._generateTokens()` and `JwtAuthGuard.canActivate()` fall back to hardcoded dev secrets:

```typescript
const secret = this.cfg.get<string>('JWT_SECRET') ?? 'dev-secret-change-me';
```

This is intentional for dev, but **no validation** that production env actually sets `JWT_SECRET`. If an operator forgets to set the env var in production, the system silently uses a weak dev secret.

**Suggested Fix**:
```typescript
private getJwtSecret(env: string): string {
  const secret = this.cfg.get<string>('JWT_SECRET');
  if (env === 'production' && !secret) {
    throw new Error('JWT_SECRET must be set in production');
  }
  return secret ?? 'dev-secret-change-me';
}
```

Then call from `_generateTokens()` and guard to validate.

### 3.4 Sync Cursor Update Race Condition (Edge Case)

**Location**: `server-cn/src/modules/sync/sync.service.ts:344-360`

**Problem**: The `upsertCursor()` method is called after `pullChanges()` completes. If two simultaneous pull requests for the same entityType arrive:
- Request A: pull (0 items), update cursor to T1
- Request B: pull (2 items), update cursor to T2
- Due to race, cursor might be T1, causing Request B's items to be re-fetched

**Suggested Fix**: Include cursor update atomically with the query:
```typescript
private async upsertCursor(
  userId: string,
  deviceId: string,
  entityType: SyncEntityType,
): Promise<void> {
  // Use ON CONFLICT for atomic update
  await this.cursors
    .createQueryBuilder()
    .insert()
    .into(SyncCursorEntity)
    .values({
      userId,
      deviceId,
      entityType,
      lastSyncedAt: new Date(),
      lastSyncedVersion: '1',
    })
    .orUpdate(['lastSyncedAt', 'lastSyncedVersion'], ['userId', 'deviceId', 'entityType'])
    .execute();
}
```

## 5. Security Posture

### Strong Points

- ✓ Bcrypt cost=12 (not default 10)
- ✓ JWT refresh token hashing (not storing plaintext)
- ✓ SMS code hashing (SHA-256)
- ✓ User-level data isolation via `userId` WHERE clauses

### Gaps

- ⚠️ No rate limiting on SMS send endpoint (`sendSmsCode()`)
- ⚠️ No CSRF protection (single-origin API, but add `SameSite=Strict` to cookies if used)
- ⚠️ AI prompt injection: User-controlled `question` field in `contentQA()` sent directly to LLM without sanitization
- ⚠️ No input validation on reading progress `percentComplete` (could accept negative or >100 values)

---

## 6. Data Integrity & Consistency

### Strengths

- ✓ Soft-delete pattern **not** used (hard deletes are explicit and intentional)
- ✓ UUID primary keys (no collision risk)
- ✓ Foreign key constraints on migrations (though not explicitly FOREIGN KEY in SQL)
- ✓ Sync conflict resolution strategy is **documented** (last-write-wins + SM-2 merge)

### Concerns

- ⚠️ Concurrent session deletion (see Critical #3) can cause phantom refresh errors
- ⚠️ Quota upsert under load (see Critical #2)
- ⚠️ No database-level constraints for user ownership (could add UNIQUE(userId, bookId) on reading_progress)
