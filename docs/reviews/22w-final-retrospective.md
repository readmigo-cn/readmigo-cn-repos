## 3. 架构整体评估

### 3.1 6 大子系统架构质量

#### 3.1.1 客户端 ArkTS (apps/harmony-app)

**架构**: UIAbility (导航层) → Pages (UI 层) → Services (业务层) → Stores (状态管理)

| 维度 | 评分 | 评价 |
|------|------|------|
| 分层清晰度 | 8/10 | ✓ 4 层分离，但 Page 与 Service 耦合过高 |
| 组件复用 | 6/10 | ⚠️ W6 组件库 (10 件套)，但新增组件仍自研 |
| 状态管理 | 7/10 | ✓ @State/@Link/@Provide/@Consume 使用规范，但循环依赖风险 |
| 错误处理 | 6/10 | ⚠️ 缺少统一的错误边界，各 Service 自行处理 |
| 性能优化 | 5/10 | ⚠️ 未验证真机帧率 / 内存压力 |

**关键问题**:
- **Reader.ets 膨胀** (1144 LOC): Canvas 渲染 + 手势 + TTS + 进度同步 + Selection 全在一个文件，应拆分为 4+ mixin
- **Service 同步队列重复**: ProgressSync/HighlightService/NoteService 有 80% 相同的 debounce/retry 逻辑，应抽象基类
- **SSE 错误处理**: ExplainCard 流式解析缺少超时 (TranslateService) 和异常恢复 (TtsService)

#### 3.1.2 后端 NestJS (server-cn)

**架构**: Controller → Service → Repository → Entity | middleware (auth/logging/error)

| 维度 | 评分 | 评价 |
|------|------|------|
| 模块化 | 8/10 | ✓ 9 个模块独立，清晰的依赖边界 |
| 数据层 | 6/10 | ⚠️ N+1 查询存在（reading.service, books.service）|
| 并发安全 | 5/10 | ⚠️ AI quota upsert 竞态、刷新令牌并发问题 |
| 错误处理 | 7/10 | ✓ GlobalExceptionFilter + 自定义异常类 |
| API 文档 | 8/10 | ✓ Swagger 装饰器完整 |

**关键问题** (W3/W7 review 确认):
1. **AI Quota 竞态**: `_incrementQuota()` 的 fallback 不使用事务，两并发请求可导致计数丢失 (line 72-86, ai.service.ts) → **P0 Critical**
2. **刷新令牌并发**: 多个请求同时收到 401，都尝试刷新令牌，可能导致令牌乱序 → **P0 Critical**
3. **N+1 查询**: `getReadingStats()` 逐个查询会话 (line 114-134, reading.service.ts)；`searchBooks()` 逐个加载作者 → **P1 High**

### 4.4 注释策略 (英文 vs 中文混合)

**现状**:
- **后端**: 90% 英文注释 + 10% 中文注释 (主要用于业务逻辑解释)
- **客户端**: 50% 中文 + 40% 英文 + 10% 无注释

**问题**:
- 关键业务逻辑（AI 配额、支付流程）用英文注释，但代码本身是中文命名 (变量名)
- SSE 流式处理、NAPI 调用等复杂代码缺少 "why" 注释

**建议**:
1. 采用"英文代码 + 中文业务注释"的混合策略
2. 关键路径（支付、数据同步、并发控制）补充 `// 📝 业务说明` 标记的中文注释

**评分**: 6/10 (缺少系统性策略)

### 4.5 代码重复率与 DRY 违反

**检测结果**:

1. **Service 同步队列重复** (ProgressSync / HighlightService / NoteService)
   ```typescript
   // 三个 service 都有这段
   private pending: T[] = [];
   private scheduleDebounce() { ... }
   private scheduleBackoffRetry() { ... }
   // 重复率: ~200 LOC / 3 = 高度重复
   ```
   
   **改进**: 抽象 `BaseSyncService<T>` 基类，包含队列管理、重试逻辑。预期可消除 150+ LOC 重复。

2. **UI 表单验证重复** (Login.ets / Register.ets / ProfileEdit.ets)
   ```ets
   // 都有这段验证逻辑
   validateEmail(email: string): boolean { ... }
   validatePassword(pwd: string): boolean { ... }
   // 可统一到 ValidationUtils
   ```
   
   **改进**: 新增 `src/utils/validation.ts` 集中管理。

3. **API 错误处理重复** (各 Service 中)
   ```typescript
   .catch(err => {
     console.error(`Failed to ${operation}:`, err);
     return null; // 或 throw
   })
   // 重复出现 ~15 次
   ```
   
   **改进**: 统一的 `ApiErrorHandler` 装饰器。

**DRY 评分**: 6/10 (存在明显改进空间，但不致命)

**预期重构收益**: 
- 代码量减少: 200-300 LOC
- 可维护性提升: 20%
- 测试覆盖提升: 15% (通过基类单测)

- 翻页响应: < 200 ms
- 内存占用 (静息): < 150 MB
- 帧率: ≥ 55 FPS (HarmonyOS 120Hz 目标)
- 包大小: < 150 MB

**实际测试** (仅模拟器):
- ❌ 冷启: 4.2 秒 (模拟器，真机未知)
- ❌ 主界面: 2.1 秒 (超预期)
- ⚠️ 翻页: 180-250 ms (边界)
- ❌ 内存: 220 MB (46% 超预)
- ⚠️ 帧率: 45-60 FPS (不稳定，缺 Profile 数据)
- ✓ 包大小: 142 MB (符合)

**评价**: 性能距离目标尚有差距，未在真机上验证。

### 6.2 已识别瓶颈

1. **Reader Canvas 渲染** (~100ms/frame)
   - 原因: 排版引擎 + Canvas 绘制在主线程
   - 建议: 考虑 Worker 并行排版 (W23+)

2. **Library 列表滚动** (FPS 不稳定)
   - 原因: LazyForEach 加载逻辑 + 书籍封面加载
   - 建议: 使用虚拟滚动、封面预加载 (W23+)

3. **AI 流式解析** (SSE 延迟 200-500ms)
   - 原因: 文本逐字符处理 + UI 更新同步
   - 建议: 批量缓冲、异步 UI 更新 (W24+)

4. **数据库查询** (N+1 问题)
   - 例: `getReadingStats()` 查询 100 条进度记录 → 100 个会话查询
   - 建议: 使用 JOIN + 批量查询，预期 10x 提速
