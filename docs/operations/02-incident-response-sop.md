## 第一部分：事件分类与定义

### 1.1 P0 事件（严重事件）

**定义：** 应用完全不可用或大范围用户受影响

**触发条件（满足任一）：**

1. **应用无法启动**
   - 症状：所有用户打开应用后立即崩溃
   - 影响：100% 用户不可用
   - 原因：严重代码 Bug、初始化失败

2. **所有用户无法登录**
   - 症状：登录接口返回错误或超时
   - 影响：新用户无法注册，老用户被登出
   - 原因：后端服务宕机、认证系统故障

3. **核心功能 100% 失效**
   - 例：所有用户无法阅读、无法朗读、无法购买
   - 影响：应用失去核心价值

4. **数据丢失 / 损坏**
   - 症状：用户书籍、笔记、阅读进度被删除
   - 影响：用户数据安全受威胁

5. **安全漏洞**
   - 症状：用户隐私数据泄露或被篡改
   - 影响：可能导致法律问题

6. **崩溃率 > 10%**
   - 症状：Sentry 显示 10% 以上用户每小时遇到崩溃
   - 影响：应用不可用概率很高

**P0 响应流程：**

```
┌─ 发现事件 (T+0 min)
│  └─ 任何人发现 → 立即通知 Incident Commander
│
├─ 组建应急队 (T+5 min)
│  ├─ Engineering Lead（应急指挥）
│  ├─ Backend Engineer（服务端）
│  ├─ Mobile Engineer（应用）
│  ├─ Ops（基础设施）
│  └─ Product Lead（决策）
│
├─ 初步诊断 (T+10 min)
│  ├─ 确认问题范围（全量用户 vs 部分用户）
│  ├─ 查看 Sentry、服务器日志
│  └─ 初步评估根本原因
│
├─ 实施修复 (T+15~45 min)
│  ├─ 方案 A（推荐）：快速 hotfix + 加急上架
│  ├─ 方案 B（降级）：临时关闭有问题的功能
│  └─ 方案 C（最后手段）：请求平台下架应用
│
├─ 验证修复 (T+45~60 min)
│  ├─ 灰度发布 10% 用户
│  ├─ 监控 5 分钟，无新 issue 则全量发布
│  └─ 验证用户反馈，确认问题解决
│
└─ 通知与总结 (T+60~90 min)
   ├─ 通知所有受影响用户
   ├─ 发布官方声明
   └─ 启动 RCA 分析
```

### 1.2 P1 事件（高优先级）

**定义：** 关键功能部分不可用或中等范围用户受影响

**触发条件：**

1. **关键功能异常**
   - 阅读卡死（但不是 100% 用户）
   - 内购无法完成（支付失败）
   - AI 朗读停止工作

2. **中等范围崩溃**
   - 崩溃率 5-10%
   - 特定设备型号无法使用（如 iPhone 15）
   - 特定用户群体 > 50% 受影响

3. **账户 / 支付问题**
   - 某些用户的订阅状态被错误更新
   - 无法登出

**P1 响应时间：** 30 分钟内组建队伍，8 小时内修复

### 1.3 P2 事件（中优先级）

**定义：** 部分用户遇到功能异常

**触发条件：**

1. **部分用户崩溃** (1-5% 的用户)
2. **功能异常但有替代方案**
3. **显示错误或性能问题**

**P2 响应时间：** 2 小时内回复，24 小时内修复

### 1.4 P3 事件（低优先级）

**定义：** 极少数用户遇到的问题

**触发条件：**

1. **崩溃率 < 1%**
2. **不影响应用核心功能**

**P3 响应时间：** 纳入日常工作流，一周内修复

### 2.1 发现与上报（T+0-5 分钟）

**谁可以触发 P0 警报：**
- Sentry 自动告警（崩溃率 > 10%）

- 应用市场评论（大量投诉）

```

【问题确认】
- 现象：应用无法启动
- 影响范围：100% 用户
- 崩溃率：Sentry 显示 95%

【后端状态】✅ 正常
- API 响应正常（/health 返回 200）
- 数据库连接正常
- 无异常错误日志

【应用状态】❌ 故障

【可能的根本原因】
```

### 2.4 修复与上架（T+15-60 分钟）

**方案 A：回滚到前一个版本（推荐，最快）**

```bash
# 前提：上一个版本可用
# 时间：5-10 分钟

# 1. 华为应用市场后台
#    版本管理 → 上一个版本 (v1.0.0 Build 0)
#    → 立即上架

# 2. iOS App Store
#    App 版本 → 上一个版本
#    → 重新发布

# 3. 告知用户
#    "已恢复到稳定版本，请更新"
```

**方案 B：快速 Hotfix（修复时间充足）**

```bash
# 1. 定位 Bug
#    示例：NullPointerException in AppInitializer.js:42
#    const data = JSON.parse(config); // config 为 null

# 2. 修复代码
# entry/src/main/ets/service/AppInitializer.ets
AppInitializer.onStart() {
  try {
    const config = this.loadConfig();
    // ❌ BUG: 若 config 为 null 会崩溃
    // const data = JSON.parse(config);
    
    // ✅ FIX: 添加 null 检查
    const data = config ? JSON.parse(config) : {};
    this.initApp(data);
  } catch (error) {
    console.error('Init failed:', error);
  }
}

# 3. 本地验证
hvigorw assemble -p product=default --mode release
# 安装到模拟器，确认 bug 已修复

# 4. 提交代码并上架
git add -A
git commit -m "fix: null pointer exception in AppInitializer"
git push origin main

# 5. 触发 CI/CD 构建 Release HAP

# 6. 华为应用市场加急上架
#    可选：申请加急审核（2-4 小时）
```

**方案 C：临时降级功能（作为最后手段）**

```bash
# 若无法快速修复代码，通过配置临时禁用有问题的功能

# 示例：如果朗读导致崩溃
# 在 RemoteConfig 中添加：
{
  "feature_flags": {
    "reading_enabled": true,  ✅
    "tts_enabled": false,     ❌ 禁用朗读以止血
    "notes_enabled": true     ✅
  }
}

# 告知用户：
# "为应对突发问题，朗读功能已临时关闭。
#  我们正在修复，预计明天恢复。抱歉！"
```

## 第四部分：关键功能降级策略

### 4.1 什么时候使用降级

**条件：**
- 无法快速修复代码（> 2 小时）
- 需要立即止血
- 降级不会影响核心功能

**示例降级场景：**

| 功能 | 故障 | 降级方案 |
|------|------|--------|
| AI 朗读 | 崩溃 | 关闭朗读功能，用户可阅读 |
| 推荐算法 | 缓慢 | 改用简单排序（热门书籍） |
| 同步 | 超时 | 禁用云同步，使用本地数据 |
| 内购 | 失败 | 向用户显示 "暂时不可用"，推荐邮件联系 |

### 4.2 如何实施降级

**使用 RemoteConfig：**

```arkts
// entry/src/main/ets/service/FeatureFlags.ets

class FeatureFlags {
  static async getFlags(): Promise<{
    tts_enabled: boolean;
    recommendation_enabled: boolean;
    sync_enabled: boolean;
    iap_enabled: boolean;
  }> {
    // 从远端配置拉取功能开关
    const response = await fetch('https://api.readmigo.cn/config/features');
    return await response.json();
  }
}

// 在 UI 中使用：
async openTTSPlayer() {
  const flags = await FeatureFlags.getFlags();
  
  if (!flags.tts_enabled) {
    AlertDialog.show({
      message: 'AI 朗读功能暂时不可用，敬请谅解。',
      buttons: [{ text: '确定' }]
    });
    return;
  }
  
  // 正常流程
  this.startTTS();
}
```

**后端立即更新配置：**

```bash
# 远端配置存储在 Neon 数据库中
UPDATE feature_flags 
SET tts_enabled = false 
WHERE app = 'harmony';

-- 立即生效，用户不需更新应用
```

## 第六部分：服务降级决策框架

### 6.1 何时考虑主动降级

**不是等用户发现，而是主动识别并降级：**

```
监控告警
  ↓
【自动决策】
  ├─ 若 3 分钟内无法恢复 → 自动降级非核心功能
  ├─ 向 Incident Commander 通知
  └─ 用户透明提示（"功能暂时不可用"）

【例：API 超时触发】
  ├─ API 响应时间 > 30s
  ├─ 同时请求数 > 10,000
  ├─ 自动启用 API 限流 (降低延迟但减少功能)
  └─ 通知用户 "系统繁忙，请稍后重试"
```

### 6.2 不同功能的降级优先级

```
【核心功能】(保护，不能降级)
✓ 应用启动
✓ 用户登录 / 登出
✓ 阅读内容呈现
✓ 用户数据持久化

【重要功能】(尽量保护，最后才降级)
⚠ AI 推荐
⚠ 笔记保存
⚠ 内购购买

【可选功能】(优先降级)
❌ AI 朗读
❌ 推荐算法
❌ 社交分享
❌ 实时同步
```

## 第八部分：特殊场景应对

### 8.1 若无法在 1 小时内修复

**决策树：**

```
能否降级？
├─ 是 → 立即启用降级，恢复部分功能
│       继续后台修复，无需下架
│
└─ 否 → 能否快速修复？
        ├─ 是 → 继续修复，预计多久？
        │       < 4 hours → 继续修复，定期通知用户
        │       ≥ 4 hours → 请求平台下架，准备 hotfix
        │
        └─ 否 → 请求平台下架或限制
                可选：回滚上一版本
                准备 hotfix 并加急审核
```

### 8.2 若用户数据被破坏

**立即行动：**

```
1. 【停止扩散】
   - 立即关闭有问题的功能
   - 阻止更多数据被破坏

2. 【数据恢复】
   - 检查数据库备份
   - 恢复至最后的好状态（通常 1 小时前）
   - 数据恢复可能需要 30 分钟 - 2 小时

3. 【用户通知】
   - 向受影响用户逐一联系
   - 解释数据丢失原因
   - 提供补偿（延长订阅、赠送内容等）

4. 【法务评估】
   - 是否涉及个人隐私数据泄露
   - 是否需要向监管部门报告（工信部、网信办）
   - 准备公开声明
```

### 8.3 若涉及安全漏洞

**立即行动（参考用户反馈 SOP 中的隐私投诉）：**

```
1. 隔离问题（15 分钟内）
2. 确认影响范围（30 分钟内）
3. 用户通知（1 小时内）
4. 公开声明（24 小时内）
5. 修复并加急上架（2-4 小时）
```
